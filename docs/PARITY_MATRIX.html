判定基準

○ 同じ契約または同じ処理方針で実装済み △ 動くが、参照製品と処理単位・入力・既定値・検証状態に差分がある × 現状は参照製品と同じロジックではない N/A v1対象外またはVideo固有処理

この表では「同じロジック」を、コード共有ではなく「同じ入力前処理、同じモデル、同じ失敗時契約、同じ公開フィールドで動くこと」として扱う。

比較表

区分 比較項目 参照製品の契約 TimelineForVideoの現状 判定 根拠 次アクション
Audio モデル・依存の棚卸し models list で pyannote、ZIPA、FFmpeg、GPU条件を確認できる。 Videoの models list --json はローカル処理と音声モデルを返す。pyannote/ZIPAは既定で必須扱いになる。 ○ model_inventory.py, Docker実行結果 なし。
Audio ffprobeメタデータとsource identity Audioは処理前に ffprobe でメディア情報を読む。 Videoはsource videoを読み取り専用で ffprobe し、source identity / fingerprintを生成する。 ○ probe.py, docs/PIPELINE.md なし。
Audio モデル入力前処理 Audioは ffmpeg で正規化WAVを作り、モデル処理は正規化WAVを読む。正規化WAVは処理中間物でmaster artifactにしない。 Videoはレビュー用MP3とは別に一時正規化WAVを作り、pyannote/ZIPAはそのWAVを読む。WAVは処理後に削除され、master artifactに残さない。 ○ Audio processor.py / Video audio_analysis.py なし。
Audio speech candidate検出 Audioは正規化WAVから候補区間を作り、候補マップを処理中だけ保持する。 Videoは一時正規化WAVを起点に ffmpeg silencedetect を実行し、候補区間を構造化証跡として保存する。 ○ trim_audio / run_silence_detect なし。
Audio speaker diarizationモデル pyannote/speaker-diarization-community-1 で機械的speaker turnを作る。 Videoも同じモデルIDを使う。依存とHugging Face tokenが揃うと実行される。 ○ audio_models.py なし。
Audio diarization必須性と失敗契約 Audioではdiarizationは必須。失敗時にfallback speakerを捏造しない。 Videoの既定は required。pyannote/ZIPAが実行できない場合はmaster完成扱いにせず、fallback speakerやphone tokenを作らない。 ○ Audio docs/PIPELINE.md / Video audio_models.py なし。
Audio ZIPA acoustic units anyspeech/zipa-large-crctc-300k をONNX Runtimeで実行し、公開フィールドは phone_tokens。 Videoも同じモデルIDと phone_tokens を使い、readable transcriptは作らない。 ○ audio_models.py, items.py なし。
Audio 長尺音声のchunk処理とspeaker割当 Audioは候補区間を小さく処理し、speaker turnとtimestamp overlapで統合する。 Videoは候補区間を最大30秒chunkに分け、正規化WAVからchunk単位で読み出してZIPAへ渡し、speaker interval overlapでspeakerを付ける。 ○ MAX_ACOUSTIC_CHUNK_SECONDS, speaker_for_interval なし。
Audio GPU fail-fast Audio GPUはCUDA/ONNX CUDA providerがなければCPUへ黙ってfallbackしない。 Videoも computeMode: gpu が既定。GPU worker、PyTorch CUDA、CUDAExecutionProvider が揃わなければ失敗し、CPUへ黙ってfallbackしない。 ○ docker-compose.gpu.yml, onnx_providers なし。
Audio フル文字起こし非対応 Audioは読みやすい文章ではなく、phone-like acoustic unitsを保存する。 Videoも readableText は空で、音声テキスト証跡は phone_tokens に限定している。 ○ audio_analysis.py, Audio docs/PIPELINE.md READMEで「音声は文字起こしではなくphone_tokens」と明記する。
Audio 生成音声artifactとZIP除外 Audioはsource audioをZIPに含めない。 Videoはsource videoも生成MP3もZIPに含めない。生成MP3は outputRoot 配下のローカルレビュー用。 ○ items.py, docs/OUTPUTS.md なし。
Image OCRエンジン ImageはローカルTesseract jpn+eng を使い、外部画像APIを呼ばない。 Videoは生成フレームartifactに対してローカルTesseract jpn+eng を使う。 ○ Image ocr.py / Video frame_ocr.py なし。
Image OCR公開フィールド has_text, full_text, block_id, bbox_norm, confidence を使う。 Videoの frame_ocr.json も同じ名前のsubpayloadを出す。 ○ frame_ocr.py, docs/OUTPUTS.md なし。
Image OCR auto失敗時の扱い Imageのauto OCRは失敗時に空OCRとwarningsを返す。 Videoもautoでは空OCRとwarningを返し、source videoやsource frameを変更しない。 ○ Image ocr.py / Video run_frame_ocr なし。
Image OCR debug overlay ImageはOCR領域のdebug overlayを生成する。 Videoも artifacts/ocr/*-ocr.jpg を生成する。 ○ write_debug_overlay, docs/OUTPUTS.md なし。
Image brightness / contrast ImageはPillowで輝度平均と標準偏差を取り、同じ閾値でdark/bright/low/highを判定する。 Videoも同じ計算と閾値を生成フレームに適用する。 ○ Image quality / Video frame_quality なし。
Image color palette Imageは160px thumbnailを MEDIANCUT 量子化して最大8色のpaletteを作る。 Videoも160px thumbnailを MEDIANCUT 量子化して最大8色のpaletteを作る。 ○ Image color_palette / Video frame_color_palette なし。
Image 3x3 color grid Imageは3x3 gridで cell_id, row, col, bbox_norm, average_color を出す。 Videoも3x3 gridで cell_id, row, col, bbox_norm, average_color を出す。 ○ Image color_grid / Video frame_color_grid なし。
Safety source fileの扱い Audio/Imageはsourceを編集せず、生成物だけをoutputRootに置く。 Videoもsource videoを変更・削除・コピーせず、生成物だけを outputRoot 配下へ置く。 ○ docs/SAFETY.md, docs/OUTPUTS.md 継続してテスト対象にする。
Safety 外部API / 人物認識 / 顔認識 Imageは外部画像APIを呼ばず、人物認識もしない。Audioはモデル実行以外の外部解析APIを前提にしない。 VideoもOCR、pyannote、ZIPA以外のscene detection、OCR外API、人物/顔認識、外部解析APIを実装していない。 ○ model_inventory.py, docs なし。
Runtime 常駐worker Audio/ImageはDocker-firstのworker/CLI製品として動く。 Videoの timeline-for-video-worker-1 は現在 Up かつ healthy。serve が定期refresh loopを持つ。 ○ docker compose ps, processor.py なし。
Runtime 実動画での現在処理状態 参照製品は対象ファイルをmaster artifactへ変換する。 Videoは常駐workerと items refresh でmaster artifactを作る。pipeline version更新により、既存itemも新しいAudio/Image parity契約で再処理対象になる。 ○ C:\TimelineData\video\items 実測 全件処理の完了待ちは運用進捗として別管理。
Runtime 一覧CLIのページング Audio/Imageは一覧UI向けに --page / --page-size を持つ。 Videoも files list, items list, runs list で --page / --page-size を受け付け、JSONに pagination を返す。 ○ cli.py, test_cli.py なし。
Repo コード共有・import禁止 Audio/Imageの機能をVideoにも実装するが、ソース共有はしない。 Video workerは独立実装。Audio/Image packageをimportしていない。 ○ 参照調査とVideo import確認 なし。
Output master artifact構造 Audio/Imageはsource別itemに timeline.json, convert_info.json, record JSONを置く。 Videoは video_record.json, timeline.json, convert_info.json, raw_outputs, artifactsを outputRoot/items/<item-id> に置く。 ○ docs/OUTPUTS.md なし。

完了した一致化

  • Audio: 一時正規化WAVをモデル入力にし、レビュー用MP3をpyannote/ZIPA入力から外した。
  • Audio: speech candidate検出を正規化WAV起点にした。
  • Audio: ZIPAは候補区間を最大30秒chunkとしてWAVから読み出す。
  • Audio: pyannote/ZIPAの既定動作を必須にした。
  • GPU: computeMode の既定を gpu にし、Audio互換モデルはCPUへ黙ってfallbackしないようにした。
  • CLI: files list, items list, runs list にページングを追加した。
  • Image: palette量子化を Image.Quantize.MEDIANCUT に揃えた。
  • Image: 3x3 gridに cell_id を追加した。
  • Docs: README、Runtime、Pipeline、Output contractを新しい処理契約に合わせた。
